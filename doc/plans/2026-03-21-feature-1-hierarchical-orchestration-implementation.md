# Feature 1 Hierarchical Orchestration Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Implement Paperclip's first issue-rooted planner/worker/verification run graph with bounded parallelism, automatic repair loops, policy-driven evidence, and review-ready issue surfaces.

**Architecture:** Extend the existing `heartbeat_runs`-based runtime into a durable run graph instead of replacing it wholesale. Keep orchestration inside Paperclip, push isolated execution into runner-compatible services, and add only the minimum runtime-bundle and evidence plumbing required to make Feature 1 real. The implementation should stay additive to the current heartbeat system so existing adapters and issue flows keep working while the new graph takes shape.

**Tech Stack:** TypeScript, Express, Drizzle ORM, PostgreSQL/PGlite, React, TanStack Query, Vitest

## Implementation Notes Before Starting

- Reuse `heartbeat_runs` as the durable run record. Add graph fields there instead of introducing a parallel `agent_runs` system.
- Keep `planner`, `worker`, and `verification` as first-class run types from the start.
- Treat `codex`, `cursor`, and `opencode` as the required runtime targets for projections. `pi` can be wired as a follow-up within the same surface; `claude` should not drive implementation choices.
- Keep runtime-bundle work minimal in this plan: just enough to satisfy worker startup context and future-proof the contract.

- the full verification command set passes
